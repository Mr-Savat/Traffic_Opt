1. PROJECT OVERVIEW
────────────────────
  We build a smart traffic signal system for a Phnom Penh-style
  intersection using QUBO (Quantum optimization math) and test it
  against traditional methods. The simulation runs in SUMO with
  70% motorcycle, 20% car, and 10% tuk-tuk traffic — reflecting
  Phnom Penh's real vehicle composition.


2. PROJECT PHASES
──────────────────

  PHASE 1 — Environment Setup
  ─────────────────────────────
  Tasks:
  • Install Python 3.11, SUMO simulator, D-Wave Ocean SDK
  • Register D-Wave Leap academic account (free access)
  • Install packages: dimod, neal, matplotlib, pandas
  • Verify all tools work correctly

  Done when:
  → Python runs, SUMO opens, dimod imports, D-Wave connected


  PHASE 2 — Build SUMO Intersection Model
  ─────────────────────────────────────────
  Tasks:
  • Design 4-way intersection in SUMO netedit
  • Configure 3 lanes per direction (12 total)
  • Set up motorcycle sublane behavior (SL2015 model)
  • Define 3 vehicle types: motorcycle, car, tuk-tuk
  • Set traffic volume: 700 moto/h, 150 car/h, 60 tuk/h
  • Add traffic light control system (TLS)

  Done when:
  → SUMO shows vehicles (70% moto) driving through intersection


  PHASE 3 — Write QUBO Optimization Code
  ─────────────────────────────────────────
  Tasks:
  • Write QUBO formula builder (motorcycle-weighted: 0.60/0.30/0.10)
  • Write Simulated Annealing solver (runs on laptop, free)
  • Write Quantum Annealing solver (D-Wave Leap cloud)
  • Write Fixed Cycle baseline controller
  • Write Webster method controller
  • Connect all code to SUMO via TraCI API
  • Add data logger (saves wait time + queue to CSV)

  Done when:
  → Traffic light changes every 30s based on actual vehicle queue


  PHASE 4 — Run Experiments
  ───────────────────────────
  We run the simulation 16 times (4 methods × 4 scenarios):

  4 Methods:
    Method 1 — Fixed Cycle        (traditional, dumb timer)
    Method 2 — Webster's Formula  (classical engineering)
    Method 3 — SA + QUBO          (our method, laptop solver)
    Method 4 — QA + QUBO          (our method, D-Wave quantum)

  4 Traffic Scenarios:
    SC-1 — Rush Hour AM   : 700 moto/hour
    SC-2 — Rush Hour PM   : 600 moto/hour
    SC-3 — Off-Peak       : 200 moto/hour
    SC-4 — Extreme Load   : 900 moto/hour

  Each run: 1 simulated hour | Saves: results_[Method]_[SC].csv

  Done when:
  → 16 CSV result files collected, SA avg_wait < Fixed avg_wait


  PHASE 5 — Analyze Results
  ───────────────────────────
  Tasks:
  • Calculate average wait time per method per scenario
  • Build Table 1: Performance Comparison (all 4 methods)
  • Build Figure 1: Bar chart — average wait time comparison
  • Build Figure 2: Line chart — queue length over time
  • Calculate improvement percentage vs Fixed Cycle baseline

  Expected key finding:
  → SA+QUBO and QA+QUBO reduce wait time ~40% vs Fixed Cycle
  → Biggest improvement in SC-4 (extreme motorcycle load)

  Done when:
  → Table 1, Figure 1, Figure 2 ready for paper
